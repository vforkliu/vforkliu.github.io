游戏安全：月影外挂透视功能分析

发表于 2019-07-06 | 更新于 2019-07-07 | 分类于 游戏安全 | 评论数：

简介

透视功能是FPS游戏面临的主要威胁之一：它可以始终显示目标的位置，极大地破坏了游戏的乐趣。实现透视涉及到两个关键技术点：目标位置计算和标注目标。

标注目标

标注目标一般画一个小方框。本文的月影外挂就是画一个小红方框。
月影外挂采用clear函数绘制小红框：

这是DirectX里的一个函数，参数含义如下：

  • Count：说明你要清空的矩形数目。如果要清空的是整个客户区窗口，则设为0；
  • pRects：这是一个D3DRECT结构体的一个数组，如果count中设为5，则这个数组中就得有5个元素。
  • Flags：一些标记组合。只有三种标记：D3DCLEAR_STENCIL , D3DCLEAR_TARGET , D3DCLEAR_ZBUFFER。
  • Color：清除目标区域所使用的颜色。
  • Z：设置Z缓冲的Z初始值。小于或等于这个Z初始值的Z值才会被改写，但它的值只能取0到1之间。
  • Stencil：设置模板缓冲的初始值。它的取值范围是0到2的n次方减1。其中n是模板缓冲的深度。 上图显示了一次Clear调用过程，画一个方框共调用四次。

目标位置计算

在3D游戏里每个游戏对象都有一个位置的成员变量，这个变量存放的是世界坐标系下的坐标。在FPS游戏里，为了游戏的流畅度和减小开发难度，敌对阵营所有目标的位置坐标在每时每刻都可以在客户端获取。或者说游戏服务器没有建立起一个三维世界，游戏服务器不知道某个游戏对象是否被遮挡，如果知道，理论下就不会下发这个游戏对象的位置信息。这是网络游戏某些逻辑呈现本地化的重要例子。

世界坐标转屏幕坐标

假设已经获取了某个目标的世界坐标，如何将世界坐标转换为屏幕坐标呢？先来看一个关键函数：

  • pOut 结果向量
  • pV 源向量
  • pViewport 视口
  • pProjection 投影矩阵
  • pView 视变换矩阵
  • pWord 世界变换矩阵
    这个函数就可以将世界坐标系的坐标转换为屏幕坐标系的坐标。

获取世界坐标

游戏客户端一般会有一个全局指针，保存了所有目标的全部信息。关键点：找到这个全局数据结构的位置。

检测方法

位置防御采用坐标加密：设计一套加密算法，对位置坐标机密，而且可以线上替换加密算法。

目标标注：根据外挂标注的方法，做对应的检测，比如检测外挂是否调用了某个函数。

图像识别：通过截屏的方式，将玩家游戏过程的游戏屏幕上传到服务器，可以采用人工标注或图像识别技术检测玩家是否作弊。

游戏安全：综述

发表于 2019-07-01 | 更新于 2019-07-06 | 分类于 游戏安全 | 评论数：

简介

本篇文章主要描述网络游戏面临哪些安全问题，出现这些安全问题的本质原因是什么，如何解决这些安全问题。

网络游戏为什么会有安全问题

网络游戏面临的大部分安全问题都是因为网络游戏部分逻辑呈现出本地化、单机化的特点导致。如果游戏所有的逻辑，资源都放在服务端是否就可以解决大部分问题呢？答案是确实可以。但是由于网络、机器性能等缘故，现阶段的游戏某些计算逻辑必须放在客户端，才可以实现流畅的游戏体验，客户端里的游戏资源更是放在客户端的。

如果5G普及，游戏客户端放在服务器运行，用户通过某种代理如浏览器进行游戏（就像现在很多移动的云测试服务），即所谓的云游戏。可能可以解决大部分问题。不过游戏运营商需要承担硬件资源，这样做，如果只是防止外挂，是否划算，需要权衡。

网络游戏具体面临哪些安全问题呢？

协议破解

通过分析游戏协议，可以将游戏里的很多操作自动化，如果完全分析清楚游戏协议，甚至都不需要游戏客户端。当然这种外挂的实现成本比较大。不过这种外挂特别适合打金工作室，因为打金工作室需要运行大量的客户端养号。

这类问题的解决主要靠一种动态协议的技术：游戏通道的关键数据加密，但解密的密钥是可以在线变化的。当然动态协议本身也提供防止包重放等功能。

UIC

用户信息控制，主要解决在游戏世界里，将游戏玩家产生的不当言论进行屏蔽。不当言论比如：政治敏感、谩骂、广告、色情信息等。

信息过滤主要还是采用特征匹配的方式完成，随着人工智能地发展，应该会引进自然语言识别相关技术。不过据笔者所知，自然语言识别还未大规模运用。

此类安全问题还包括：色情图片识别，比如玩家头像等。

资源破解

比如游戏里的珍贵皮肤、武器等，通过替换资源，自己就可以免费使用这些皮肤、武器等。这类安全问题本身不会给玩家带来收益，但是这些使用资源被替换的玩家，将游戏过程录制成视频，进行传播，会游戏口碑造成极坏地影响。

还有一种类型，可以给玩家带来极大地收益。比如某个比较难的关卡，通关后可以获得很丰厚的奖励，如果玩家将这个关卡的地图替换为比较简单的地图，就会比较容易通关，毕竟碰撞检测的逻辑都是放在本地的。

这类问题主要通过文件校验地方式解决。

按键精灵

按键精灵是实现自动化操作的主要方法。通过将游戏的某些操作任务，领金币等利用按键精灵自动化，也能为玩家带来极大地收益。这也是打金工作室比较喜欢的方式。

按键精灵对三消类的游戏也危害极大。

这类问题主要通过检测按键精灵本身来解决。有行业相关人员采用机器学习地方式将玩家的操作序列进行分析，试图识别出某些操作序列是否是自动化完成的，但效果不好。

打金工作室

打金工作室通过各种手段养号、攒游戏里的金币、稀有物品等进行售卖，会严重破坏游戏的经济系统。

这类问题主要通过数据分析来解决：如账号是有聚集性比如IP地址、机器配置、玩家等级，持有的装备等。

透视

这是FPS类游戏主要的作弊方式，实现透视需要解决两个问题：人物坐标计算和人物显示，显示的方式可以拦截每一帧绘制结束时，在将自己的绘制指令插入进去。当然还有通过关闭深度测试等方法。后续会有文章专门分析一款透视外挂的实现方式。

解决这类问题的主要方法就是坐标加密，不过要与游戏性能取得平衡。

自动开枪

简单地讲就是当敌人出现时，自动瞄准自动开枪，这也是FPS类游戏的一大痛点。自动开枪部分是通过模拟按键实现。

解决这类问题：检测模拟按键类软件。

无后座

FPS类游戏的枪械如果无后座力，开枪击杀敌人的准确率将大大提高。

这类问题主要通过收集开枪时枪械的属性数据，通过后端数据分析来完成检测。

无烟雾

FPS类经常会有烟雾弹类的道具，通过外挂，可以无视烟雾。

这类问题的解决方法与无后座差不多。

上帝视角

也可以叫无限视距，这类外挂可以使MOBA类游戏中的所有敌人都在你的视野里，即使敌人在草丛里。之所以可以实现这类外挂，本质上是坐标逻辑的计算放在本地。

这类外挂的解决方法视外挂的实现方法而定。后续会有文章详细阐述。

私服

这并不算外挂的范畴，但也对游戏的生命周期造成极大的危害，最著名的私服莫过于魔兽世界的开源服务器，据本人了解，国内的私服基本上是源码泄漏或者二进制文件泄漏搭建的，很少有通过分析游戏协议，自己动手写游戏服务器。

解决方法：法律。

游戏漏洞

比如某些任务可以无限刷、在MOBA类游戏里通过发送某个数据包直接可以让对方投降，这类安全问题称为游戏漏洞，一旦在线上版本被玩家发现，将造成巨大的损失，及其严重。

一般会成立游戏漏洞挖掘团队，尽可能提前发现这些危害巨大的漏洞。

最后的防线

法律。如果所有的技术手段都无法解决，或者解决成本过高，可以让法务解决。

游戏安全：维护PC端壳的一些事儿

发表于 2019-06-30 | 更新于 2019-07-06 | 分类于 游戏安全 | 评论数：

笔者维护壳的时间大约在2011年-2013年这两年间，时间比较久远，很多事情都不记得了，唯一地感觉就是累，一个人维护所有游戏的加壳。在那段时间经常在凌晨两三点钟被电话叫醒，然后打车去公司处理突发情况。

如何写一个针对PE文件的壳，可以参考《加密与解密》这本书。想一想当时为什么这么多问题？笔者维护这个壳除了经典的壳功能，比如压缩、导入表加密等（为游戏提供基础的安全保护），还负责其他安全组件的加载，并保证它们的正常运行。常见的安全组件包括：特征扫描引擎、云查引擎、协议安全模块等。

维护壳经常遇到的问题：某个游戏加壳之后，不能正常运行或者游戏表现不正常，没加壳，一切正常，那么一定是壳导致。这个逻辑其实没错，但是加壳之后，还启动很多安全组件，一旦出问题，需要壳的维护者确认是否是壳的问题，还是其他安全组件的问题。排查问题的方法也比较简单：二分法。比如一共启动了10个安全组件，让其只加载5个安全组件，看是否有问题，来定位是哪个安全组件导致的。

这里记录两个具体示例：游戏花屏和无法登陆。

游戏花屏

问题描述：测试同事在某台机器上运行加壳后的游戏，游戏花屏。只在一台测试机上有问题，其他测试机器正常。

问题分析：看到问题描述，感觉像是壳与特定的机器环境相互作用导致的。还是老办法：二分法。一个组件一个组件的加载运行，最后发现单独加载某个组件或者某两个组件，都不会出现花屏，单加壳也不会出现。难道是几个特定的组件放在一起才会有问题？

最终结论：每个安全组件和壳本身都没有问题，只是那台测试机器配置太低，加壳后会多占有内存导致的，如果不加壳，写个一个DLL什么事也不干，只是分配安全组件占用的总内存，注入到没加壳的游戏进程中，也会花屏。

无法登陆

问题描述：某个游戏需要阻止一个特定模块的注入，因为这个模块的注入导致其他安全组件的逻辑异常，而这个特定模块使用也很大。因为壳拦截了类似LoadLibrary的函数，只需要将要阻止的模块名字加入到黑名单即可。然后对该游戏加壳，经过测试，确实可以阻止目标模块的注入。但是并没有仔细测试，加壳后的游戏不能登陆，就是无法联网了。虽然有开发，测试一系列流程，还是将有问题的游戏版本放到外网了，可以有时候后续的测试流程也是个摆设。

原因分析：阻止了目标模块的注入，为什么该游戏就无法联网了呢，其他程序如浏览器都是可以正常访问网络的啊。其实目标模块是LSP链上的一个模块，如果该模块无法加载，相对于游戏进程来说，LSP链是不正常的，故无法连网。

解决办法：使用命令或者一些工具，可以对本机重置LSP链，就是所以第3方LSP模块统统不加载，即可正常连网。幸亏是个DAU不大的游戏，且并不是所有机器都存在这个特定的模块。

概念

程序是一系列指令的集合，通常存为可执行文件。进程是程序的执行过程。

GID:进程组ID

孤儿进程

使用fork创建的子进程，父进程应该调用wait函数，等待子进程的退出信息，如果父进程先结束，子进程将变成孤儿进程，过继给init进程，

进程组

进程组包含多个进程，进程组会有一个领导进程(process group leader)，该领导进程的PID将作为进程组的PGID。

JVM:内存区域和内存溢出异常

发表于 2019-06-21 | 更新于 2019-06-24 | 评论数：

运行时内存区域

  • Program Counter Register:程序计数器，线程私有，不会发生OOM的区域。当执行Java方法时，指向正在执行的虚拟机字节码指令的地址，当执行
    Native方法时，它的值是undefined。
  • JVM Stacks: Java虚拟机栈。线程私有。存局部变量、对象引用和returnAdress(指向一条字节码指令的地址)。
  • Native Method Stack: 本地方法栈。Java虚拟机为执行本地方法提供的栈。
  • Java Heap: Java堆是垃圾收集器管理的主要区域。线程间共享。
  • Method Area: 线程共享区域。存已加载的类型信息、常量、静态变量和即时编译器编译后的代码等数据。
  • Runtime Constant Pool: 运行时常量池是方法区的一部分。存编译期生成的各种字面量和符号引用。
  • Direct Memory: 不是Java虚拟机运行时数据区的一部分。JDK1.4引入的NIO，使用Native函数库直接分配堆外内存。

对象访问

  • Java VM Stack: “Object obj” 是一个reference类型数据，存放在Java栈的本地变量表中。
  • Java VM Heap: “new Object()” 是一个对象实例，存放在Java堆上。
  • Java Method Area: 对象的类型、父类、实现的接口和方法存在方法区。
